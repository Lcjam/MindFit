1. 개요 · 08 대체 범위

이 문서는 docs/architecture/08_system-architecture.html의 스키마 섹션을 대체한다. 08 문서 본문은 이력 보존을 위해 남기되, 상단에 이 문서로의 대체 안내 배너를 추가한다 (§12 후속 문서 갱신).

08 대비 변경점 3가지
① specialties·approach_types·session_types JSON 컬럼 → 마스터+매핑 테이블로 정규화 (ADR-1)
② 예약 동시성 제어: sessions에 active_slot NULL 트릭 + 유니크 제약 추가 (ADR-2)
③ 신규 엔티티 6종: refunds·counselor_penalties·disputes 및 정규화 매핑 테이블 3종

2. 엔티티 관계 개요

Phase별 소유 도메인과 핵심 관계. 화살표는 FK 방향(N → 1).

users 1─1 counselor_profiles
counselor_profiles N─N specialties      (경유: counselor_specialties)      ADR-1
counselor_profiles N─N approach_types   (경유: counselor_approach_types)   ADR-1
counselor_profiles 1─N counselor_session_types                            ADR-1
counselor_profiles 1─N counselor_credentials
counselor_profiles 1─N counselor_availabilities
counselor_profiles 1─N counselor_penalties                                ADR-7

sessions N─1 counselor_profiles · N─1 users(client)
sessions 1─1 questionnaires
sessions 1─1 payments
sessions 1─1 chat_rooms
sessions 1─1 disputes (nullable)
sessions 1─1 reviews (nullable, COMPLETED 이후)

payments 1─N refunds                                                      ADR-5
chat_rooms 1─N messages                                                   ADR-9/10
users 1─N notifications

4. 설계 원칙 · 패턴 준수

  • 기존 패턴 유지: BusinessException + ErrorCode, ApiResponse<T>, @PrePersist 타임스탬프
  • 수수료 스냅샷(payments.commission_rate/amount/net_amount)은 의도적 비정규화 — 요약 재계산 대신 정산 시점 값 고정
  • 예약 동시성은 락이 아닌 DB 유니크 제약(ADR-2) — 코드 0줄로 레이스 차단, 서버 스케일아웃에도 유효
  • 정규화는 다대다 관계에만 적용(ADR-1) — 값이 2개뿐인 session_types까지 마스터 테이블화하는 과설계는 기각
  • P2 시작 시 Flyway 도입 — 매핑 테이블·제약·시드 데이터 증가로 ddl-auto 의존 한계 도달
