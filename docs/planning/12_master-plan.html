3. 아키텍처 결정 기록 (ADR)

전제: 단일 서버 + 단일 MySQL MVP, 개발자 1인 + AI 에이전트, TDD 필수. Redis/Kafka/MQ 등 인프라 추가 없음. 오너 결정 표시는 2026-07-07 확정.

ADR-0. 세션 상태머신 — AWAITING_CONFIRM 분리 오너 결정
위치: session/Session.java (상태 enum은 P3에서 전체 확정 — 이후 Phase의 enum 재작업 방지)
PENDING_PAYMENT → (결제완료) → AWAITING_CONFIRM
  ├─ 상담사 수락 → CONFIRMED → COMPLETED
  ├─ 상담사 거절 → REJECTED + 전액환불
  ├─ 24h 미응답  → 자동 REJECTED + 전액환불 (스케줄러, paid_at 기준)
  └─ 취소 → CANCELLED_BY_CLIENT / CANCELLED_BY_COUNSELOR / NO_SHOW

근거: 04 요구사항의 "예약 확인/거절(Must) + 24h 이내 응답 권고 + 거절 시 자동 전액 환불"과 정합. 기존 CLAUDE.md의 "웹훅 → session.status = CONFIRMED" 규칙은 이 결정으로 폐기·수정된다 (§12).

ADR-1. specialties/approach_types 정규화 — 마스터 + N:M 매핑
위치: counselor/ 패키지 (Specialty, CounselorSpecialty, ApproachType, CounselorApproachType, CounselorSessionType)

JSON 컬럼 3개 폐기. specialties·approach_types는 마스터 테이블 + 매핑 테이블(관리자 확장 가능, 시드 데이터). session_types는 값이 CHAT/VIDEO_LINK 2개뿐이라 마스터 없이 ENUM 매핑 테이블. 검색은 (specialty_id, counselor_id) 역방향 인덱스 + JPQL EXISTS — QueryDSL 미도입(필터 3종에 과설계).

ADR-2. 예약 동시성 — DB 유니크 제약 + active_slot NULL 트릭
위치: session/Session.java, SessionService.java

UNIQUE(counselor_id, scheduled_at, active_slot) — 활성 예약은 active_slot='Y', 취소/거절 시 NULL(MySQL 유니크 인덱스는 NULL 중복 허용 → 슬롯 자동 반환). 서비스는 사전 체크(친절한 409) + DataIntegrityViolationException → SESSION_TIME_CONFLICT 변환. 비관적 락은 락 대상 부자연·테스트 곤란, 분산 락은 인프라 위반으로 기각. 슬롯이 1시간 단위 고정이라 범위 겹침 문제 없음 — 유니크 제약이 정확히 들어맞고 서버 2대가 돼도 유효.

ADR-3. 문진표 접근 매트릭스 (기존 단일 규칙 폐기)
위치: questionnaire/QuestionnaireAccessPolicy.java (순수 자바 — 단위 테스트 용이). 컨트롤러는 인증만, 판정은 서비스 레이어(08의 이중 검증 원칙 유지)
요청자조건허용
ROLE_CLIENT본인 작성분조회 항상. 수정은 결제 완료 전(PENDING_PAYMENT)까지
ROLE_COUNSELOR본인 세션 + isAccessible=true조회만 (결제 전 차단 유지)
ROLE_ADMIN항상 (분쟁 대응)조회만 + 접근 감사 로그(PII 마스킹)
그 외—403 (404 위장 없이 403 통일)

isAccessible의 의미를 "상담사 경로에만 적용되는 게이트"로 축소. CLAUDE.md 규칙 수정 대상 (§12).

ADR-4. 결제 confirm — 이중 금액 검증 + 멱등 + 후처리 공유
위치: payment/PaymentService.java, TossPaymentsClient.java, PaymentCompletionProcessor.java
  1. orderId로 조회 → 없으면 404
  2. 멱등: COMPLETED + 동일 paymentKey → 재처리 없이 200 / paymentKey 불일치 → 409
  3. 1차 검증 (토스 호출 전): 요청 amount ≠ 저장 amount → 400, 토스 미호출 (위변조 차단)
  4. 토스 승인 API 호출
  5. 2차 검증 (응답): 응답 totalAmount ≠ 저장 amount → 즉시 취소 API로 승인 롤백 + FAILED
  6. 후처리는 PaymentCompletionProcessor 단일 트랜잭션 — 웹훅과 confirm이 공유. 먼저 도착한 쪽이 처리, 나중 것은 멱등 스킵. 내용: payment COMPLETED + session AWAITING_CONFIRM(ADR-0) + ChatRoom 생성 + isAccessible=true

결제 initialize의 금액은 서버가 상담사 가격에서 계산 — 클라이언트 입력 무시. 웹훅 응답은 항상 200(토스 재시도 폭주 방지), 서명 불일치만 401.

ADR-5. 환불 — refunds 상태머신 테이블 + 자동 재시도 (통합 판정)
위치: payment/Refund.java, RefundService.java, RefundPolicy.java, RefundRetryScheduler.java
REQUESTED → IN_PROGRESS → COMPLETED
                        └→ FAILED_RETRYABLE → (지수 백오프 재시도, 최대 5회)
                        └→ FAILED_MANUAL   → 관리자 이메일 + 수동 강제환불 경로
  • 취소/거절 발생 시 로컬 트랜잭션으로 refunds INSERT(+UUID idempotency_key) + 세션 상태 변경 — 트랜잭션 내 외부 HTTP 호출 금지
  • 커밋 직후 동기 1차 시도. 토스 취소 API에 Idempotency-Key 전달 → 재시도 시 이중 환불 원천 차단. 재시도 전 토스 조회 API로 기취소 여부 선확인
  • 테이블이 곧 재시도 큐이자 환불 감사 로그 — MQ 불필요, 재시작에도 유실 없음
  • 충돌 판정 기록: API 플래너는 "자동 재시도는 과설계, 관리자 수동 재호출"을 제안했으나, 24h 자동거절 스케줄러가 P6에 어차피 필수(자동거절→전액환불을 사람이 지켜보지 않음)라 재시도 스케줄러의 한계비용이 낮아 자동 재시도를 채택
ADR-6. 스케줄러 — @Scheduled 4종, ShedLock 미도입
위치: 각 도메인 패키지. common/config/SchedulingConfig.java(@EnableScheduling, 테스트 프로파일 비활성)
잡주기동작
SessionAutoRejectScheduler5분AWAITING_CONFIRM + paid_at ≤ now−24h → REJECTED + 전액 refunds + 이메일
SessionAutoCompleteScheduler10분CONFIRMED + 종료시각 경과 → COMPLETED
RefundRetryScheduler1분ADR-5 재시도
PaymentReconciliationScheduler10분PENDING 30분 경과 → 토스 조회 대조 → 완료 동기화(웹훅 유실 보완) 또는 FAILED+슬롯 반환

스케줄러는 "스캔+루프"만, 건별 처리는 도메인 서비스의 독립 트랜잭션(1건 실패가 배치 전체를 안 깨뜨림). ShedLock은 단일 인스턴스 + 전 잡 멱등이라 불필요 — 스케일아웃 시 백로그.

ADR-7. 상담사 귀책 — B안: 전액환불 + 페널티 누적 + 노출 제외 오너 결정
위치: session/SessionCancelService.java, counselor/CounselorPenaltyService.java, admin/AdminSessionController.java

거절·당일취소·노쇼 전부 내담자 전액 환불. 당일취소/노쇼는 counselor_penalties 기록 + penalty_count 증가, 30일 내 3회 누적 시 검색 목록 자동 제외(노출 플래그만 off) + 관리자 알림 → 수동 해제/정지 결정. 금전 페널티 없음(정산 복잡도 0). 노쇼 판정은 자동화하지 않음 — 내담자 신고(report-no-show) → 관리자 분쟁 해결 경로 (§5 P6 disputes).

ADR-9. 채팅 송신 — REST POST + STOMP 수신 전용 오너 결정
위치: chat/ (BE: REST 컨트롤러 + SimpMessagingTemplate), features/chat/ (FE)
  • 송신 POST /chat-rooms/{id}/messages — P1의 401 큐잉·재시도·에러 UI 인프라 그대로 재사용. STOMP publish의 fire-and-forget 실패 감지(RECEIPT/타임아웃)·토큰 만료 재연결 경로를 만들 필요 없음
  • STOMP는 /topic/chat/{roomId} 브로드캐스트 구독 전용. 채널 인터셉터는 CONNECT/SUBSCRIBE 검증만
  • 중복 제거: 클라이언트 clientMessageId(UUID) 생성 → POST body·브로드캐스트 DTO 양쪽 포함 → mergeMessages() 순수 함수가 POST 응답↔에코를 단일 항목으로 수렴. 실패 시 pending을 failed 표시 + 재전송
  • 상태 배치: TanStack Query = 이력(useInfiniteQuery), Zustand chatStore = 연결상태·방별 라이브 버퍼·안읽음 뱃지
  • 기존 CLAUDE.md의 "전송 /app/chat/{roomId}/send" 규칙은 폐기·수정 (§12). 07 기술스택의 Optimistic Update 서술은 HTTP 뮤테이션이 되면서 다시 유효
ADR-10. 채팅 페이징·가상화 — 커서 기반, 가상화 MVP 미도입 (통합 판정)
위치: features/chat/MessageList.tsx (교체 반경 격리), useChatMessages.ts

이력 조회는 ?before=<messageId>&size=30 커서 기반 — 오프셋은 새 메시지 유입 시 페이지 경계 중복/누락 발생 (API 플래너의 page&size 초안을 FE 플래너 권고로 교체). 스크롤 앵커링은 prepend 직전 scrollHeight 기록 + useLayoutEffect 보정(overflow-anchor 브라우저 신뢰 불가). 가상화는 1:1 상담 채팅 특성상 메시지 수 유계라 미도입 — 방당 로드 누적 ~1,000건 또는 실측 jank 시 @tanstack/react-virtual 도입. MessageList를 "정렬된 배열을 그리기만 하는" 컴포넌트로 격리해 교체 시 파일 1개만 수정.

ADR-11. 토큰 저장 — MVP는 localStorage 유지 + 저비용 완화
위치: (MVP 이후) authStore.ts, api/axios.ts, api/auth.ts 3개 파일로 영향 국한

트레이드오프 명시: XSS 성공 시 refresh(7일) 탈취 가능. 그러나 F5 유지·401 큐잉·서버 저장값 대조가 이미 테스트 통과 상태이고, httpOnly 이관은 BE+FE 동시 변경으로 크리티컬 패스 지연. MVP 완화책(P5): linkify http/https 스킴만 허용(javascript: 차단), dangerouslySetInnerHTML 금지, 배포 시 CSP 헤더. MVP 이후: refresh만 httpOnly·Secure·SameSite=Lax 쿠키 + access는 메모리 + 부팅 silent refresh (BE 0.5~1일 + FE 0.5일).

ADR-12. FE 폴더 구조 — 현행 유지 + features/ 레이어 절충 (FSD 미도입)
위치: frontend/src/features/ 신설. frontend/CLAUDE.md에 승격 규칙 반영 (§12)

11. V1 / V2 로드맵 조정

V1 — 운영 품질 (재산정: 12~15 작업일, 보수 기준 캘린더 5~6주)

기존 09의 V1 항목 중 취소·환불 자동화, 승인/거절 이메일은 이번 재설계로 MVP에 편입 완료 — V1에서 "완료 처리"로 정리해 이중 계상 제거.
  1. 이메일 인증 + 카카오 OAuth — 04 요구사항 Must였으나 P1 미구현 범위 부채. V1 최우선 승격 (04 문서 MoSCoW 갱신 필요)
  2. 토스 라이브 전환 (R7 — 사업자/PG 계약 리드타임 포함)
  3. 반응형 웹 (모바일 브라우저 주요 화면)
  4. 관리자 통계·분쟁 처리 고도화·회원 관리 (정지/탈퇴)
  5. 정산 계좌 등록 — 계좌 실명조회 API는 형식 검증으로 축소 권고
  6. 리뷰 답글·수정/삭제 · 성능 튜닝 (N+1·인덱스)
  7. 토큰 httpOnly 이관 (ADR-11 로드맵 — BE 0.5~1일 + FE 0.5일)

V2 — 성장·확장 (재산정: 8~10 작업일, 기존 8주 → 2~4주 압축)

기존 항목조정근거
React Native 앱PWA로 대체 (매니페스트 + 웹푸시)1인 개발에서 RN 신규 스택 + 스토어 심사 + 인앱결제 정책(상담 결제와 Apple IAP 충돌 소지) 3중 리스크. 실사용 지표 확보 후 재검토
AI 매칭유지·V2 선두 승격, 규칙 기반으로 축소문진표 기반 가중치(전문분야+가격+평점)까지만. A/B 구조는 트래픽 부재로 제거. 포트폴리오 가치 높음
구독 플랜보류단건 결제 검증 전 빌링키 연동 복잡도만 추가
B2B EAP보류 (백로그)기업 고객 없는 선행 투자 — 검증 불가
전문분야 확장유지마스터 테이블(ADR-1) 덕에 시드 추가 수준

KPI 교체: "상담사 30명/월 세션 100건"(검증 불가 사업 KPI) → 시연 가능성 KPI: 3역할 풀 시나리오 데모, PWA 설치+푸시 수신 시연, AI 매칭 추천 근거 표시.
