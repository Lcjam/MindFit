1. 전체 스택 개요 (Stack Overview)

2. 백엔드 상세 (Backend)

Spring Boot 3.x 핵심 프레임워크 · 확정

JVM 생태계 성숙도, 팀 확정 스택. Spring Security·JPA·WebSocket 등 하위 모듈과의 완벽한 통합 지원. Jakarta EE 네임스페이스 기반의 최신 LTS 버전.

  • Java 21 기반 (LTS), Spring 6.x 코어 사용
  • Auto Configuration으로 빠른 프로젝트 셋업
  • Actuator를 통한 헬스체크·메트릭 제공 (운영 대비)
Spring Security 6 인증·인가

JWT 기반 Stateless 인증, 역할(ROLE_CLIENT / ROLE_COUNSELOR / ROLE_ADMIN)별 엔드포인트 접근 제어.

  • Access Token 유효기간: 30분 / Refresh Token: 7일
  • Refresh Token은 HttpOnly Cookie로 전송 (XSS 방어)
  • @PreAuthorize 어노테이션으로 메서드 수준 인가
  • STOMP CONNECT 프레임 헤더에서 JWT 검증 (ChannelInterceptor 적용)
  • CORS: CorsConfigurationSource Bean으로 localhost:5173 허용 (개발 환경)
Spring Data JPA + Hibernate ORM

객체-관계 매핑(ORM)으로 생산성 향상. Repository 추상화로 CRUD 코드 최소화.

  • N+1 문제 주의: 연관 엔티티 조회 시 @EntityGraph 또는 fetch join 사용
  • 복잡한 조회 쿼리는 JPQL / QueryDSL 병행
  • 인덱스 활용을 위한 네이티브 쿼리 옵션 열어둠
주의: 상담사 목록 조회, 메시지 이력 페이지네이션 등 고빈도 조회 API에서 N+1 문제가 발생하기 쉬움. 개발 단계부터 Hibernate 쿼리 로그 확인 필수.
Spring WebSocket + STOMP 실시간 채팅

실시간 채팅을 위한 WebSocket 서버. STOMP는 수신 전용(구독)이며, 송신은 REST POST로 처리한다 (12번 마스터 플랜 ADR-9 — /app/... 전송 경로는 폐기).

  • 엔드포인트: /ws (SockJS 호환 fallback 포함)
  • 구독: /topic/chat/{roomId} (수신 전용). 송신은 POST /api/v1/chat-rooms/{id}/messages — 저장 후 서버가 SimpMessagingTemplate으로 동일 토픽에 브로드캐스트
  • 채팅방은 결제 완료 후처리(PaymentCompletionProcessor)에서 서버가 자동 생성
  • 메시지 수신 즉시 MySQL messages 테이블에 저장
  • SockJS HTTP 폴백 시 WebSocket 핸드셰이크 헤더 사용 불가 → STOMP CONNECT 프레임 헤더(Authorization: Bearer {token})로 인증
  • 서버: ChannelInterceptor에서 CONNECT 커맨드 수신 시 토큰 검증
  • 쿼리 파라미터(?token=xxx) 방식 금지 — 서버 액세스 로그에 JWT 노출
Spring Mail 이메일 발송

외부 이메일 서비스 API 없이 SMTP로 직접 발송. JavaMailSender 추상화 활용.

  • 이메일 인증 링크 발송 (회원가입)
  • 상담사 자격 심사 승인/거절 알림
  • 세션 예약 확정·취소 알림
Gradle (Groovy DSL) 빌드 도구

Spring Initializr 기본값, 의존성 관리 및 멀티모듈 구성에 유연.

  • Maven 대비 빠른 빌드 캐싱
  • build.gradle (Groovy DSL) 기준으로 운영
spring-dotenv (me.paulschwarz:spring-dotenv:4.0.0) 시크릿 관리

로컬 개발 시 backend/.env 파일을 Spring 환경 변수로 자동 로드. application-dev.yml에 평문 시크릿 없이 ${env.VAR_NAME} 참조 방식 사용.

  • backend/.env — 실제 시크릿 값 (gitignore 적용, 커밋 금지)
  • backend/.env.example — 변수 목록만 (값 없음, 커밋 포함)
  • application-dev.yml, application-prod.yml 모두 gitignore 적용
  • CI/CD 환경에서는 시스템 환경변수 또는 Secret Manager 사용

대안 비교 — Spring MVC vs WebFlux

항목 Spring MVC (선택) Spring WebFlux
처리 모델 동기 / 스레드 블로킹 선택 비동기 / 논블로킹 (Reactor)
팀 친숙도 높음 (일반적인 Java 백엔드 경험 적용) 낮음 (리액티브 패러다임 러닝커브)
JPA 호환성 완벽 호환 R2DBC 필요, JPA 직접 사용 불가
디버깅 용이성 스택 트레이스 직관적 Reactive 스택 트레이스 복잡
WebSocket 지원 Spring WebSocket STOMP (충분) Reactive WebSocket (오버스펙)
선택 이유 MindFit의 트래픽 규모에서 MVC 성능으로 충분. 팀 친숙도와 JPA 통합을 우선. WebFlux의 복잡성을 피하고 빠른 개발 속도 확보.

3. 데이터베이스 상세 (Database)

MySQL 8.x 확정 DB

심리상담 플랫폼은 사용자-상담사-세션-결제-문진표 간의 복잡한 관계형 데이터가 핵심. MySQL의 트랜잭션 보장(ACID), 성숙한 생태계, Spring Data JPA와의 완벽한 호환성으로 선택.

  • Window Function, CTE (Common Table Expression) 지원 (8.x)
  • JSON 컬럼 지원 — specialties, session_types, approach_types 저장에 활용
  • Full-Text Search (상담사 검색 보완에 활용 가능)
  • 로컬 개발: Docker Compose로 MySQL 컨테이너 실행
  • 운영: AWS RDS MySQL 8.x (Multi-AZ 추후 검토)

핵심 테이블 목록

사용자 & 상담사
  • users — 회원 기본 정보, 역할, 상태
  • counselor_profiles — 상담사 프로필, 전문 분야, 가격
  • counselor_credentials — 자격 서류 심사
  • counselor_availability — 가용 시간 (주간/특정일)
상담 & 결제
  • sessions — 예약/세션 상태 관리
  • questionnaires — 문진표 (결제 후 열람)
  • payments — 결제·수수료·환불 내역
채팅 & 소통
  • chat_rooms — 결제 완료 시 자동 생성
  • messages — 실시간 채팅 메시지 영속
  • notifications — 시스템 알림
평가
  • reviews — 세션 완료 후 내담자 리뷰

인덱스 전략 포인트

  • 상담사 검색: counselor_profiles(status, price_per_session) 복합 인덱스, JSON 컬럼 함수 인덱스 검토
  • 세션 조회: sessions(counselor_id, status), sessions(client_id, status) 각각 인덱스
  • 메시지 이력: messages(chat_room_id, created_at) 복합 인덱스로 페이지네이션 성능 확보
  • 알림: notifications(user_id, is_read, created_at) — 안 읽은 알림 빠른 조회
  • 결제 검증: payments(pg_transaction_id) 유니크 인덱스 (중복 웹훅 방어)

대안 비교 — MySQL vs PostgreSQL

항목 MySQL 8.x (선택) PostgreSQL 16
팀 친숙도 높음 선택 중간
JSON 지원 JSON 컬럼 (부분 인덱싱 제한) JSONB (강력한 인덱싱·연산)
JPA 호환성 완벽 완벽
AWS RDS 지원 RDS MySQL / Aurora MySQL RDS PostgreSQL / Aurora PostgreSQL
전환 시점 JSON 컬럼 활용 빈도가 매우 높아지거나 복잡한 집계 쿼리가 필요해지면 PostgreSQL 전환 검토 가능. 현재는 MySQL로 확정.

4. 프론트엔드 상세 (Frontend)

React 18 + TypeScript + Vite UI 기반

React 18의 Concurrent Features(Suspense, Transition), TypeScript의 컴파일 타임 타입 검사, Vite의 ESM 기반 초고속 HMR로 개발 생산성 확보.

  • Strict Mode 활성화로 잠재적 버그 조기 발견
  • Vite 빌드: esbuild 기반 번들링, 운영 빌드 최적화
  • Tailwind CSS v4 — @tailwindcss/vite 플러그인 방식 (tailwind.config.ts 불필요)
  • 개발 서버 proxy: /api → http://localhost:8080 (CORS 우회 없이 직접 포워딩)
TanStack Query v5 (React Query) 서버 상태 관리

REST API 호출의 로딩·에러·캐싱·재요청을 선언적으로 관리. Redux 없이 서버 상태 처리 가능.

  • 자동 백그라운드 리패치 (상담사 목록, 세션 상태 최신화)
  • Optimistic Update — 메시지 전송은 POST /chat-rooms/{id}/messages REST 뮤테이션(TanStack Query) 기준. WebSocket(STOMP)은 수신 전용이며 송신에는 사용하지 않는다 (12번 마스터 플랜 ADR-9, 2026-07-07 확정)
  • Query Invalidation — 결제 완료 후 채팅방·세션 데이터 갱신
Zustand 클라이언트 전역 상태

인증 토큰, 채팅 알림 뱃지, 현재 사용자 정보 등 경량 전역 상태 관리. Redux 대비 보일러플레이트 최소화.

  • Auth Store: Access Token, User 정보, 로그인 상태
  • Notification Store: 읽지 않은 알림 카운트, WebSocket 수신 알림
React Router v6 라우팅

클라이언트 사이드 라우팅 및 역할별 Protected Route 구현.

  • 비인증 사용자 → 로그인 페이지 리다이렉트
  • 역할 불일치 → 403 페이지 또는 홈 리다이렉트
  • Outlet 기반 레이아웃 중첩 라우팅
Tailwind CSS 스타일링

유틸리티 클래스 기반 스타일링으로 컴포넌트별 CSS 파일 관리 부담 제거. 빠른 UI 프로토타이핑.

  • 디자인 토큰 (색상, 간격)은 tailwind.config.ts에서 중앙 관리
  • 반응형 디자인 유틸리티 내장 (md:, lg: 접두사)
React Hook Form + Zod 폼 검증

비제어 컴포넌트 기반 폼 관리로 리렌더링 최소화. Zod 스키마로 런타임 타입 검증.

  • 적용 폼: 회원가입, 로그인, 프로필 수정, 문진표, 리뷰 작성
  • 백엔드 API 에러 메시지를 폼 필드에 매핑
  • Zod 스키마를 TypeScript 타입으로 재사용 (z.infer<>)
SockJS + @stomp/stompjs WebSocket 클라이언트

브라우저 WebSocket 지원이 불완전한 환경에서 SockJS가 HTTP Long-polling 등으로 자동 폴백. STOMP 클라이언트는 구독(수신)만 담당하며, 메시지 전송은 REST POST로 별도 처리한다 (ADR-9).

  • 채팅방 입장 시 /topic/chat/{roomId} 구독
  • 메시지 전송: POST /api/v1/chat-rooms/{id}/messages (TanStack Query mutation, clientMessageId 포함)
  • Zustand 스토어와 연동해 수신 메시지를 전역 상태에 반영

대안 비교 — Redux Toolkit vs Zustand

항목 Zustand (선택) Redux Toolkit
보일러플레이트 매우 적음 선택 Slice·Selector·Action 필요
학습 곡선 낮음 중간
DevTools 미들웨어 추가 시 사용 가능 Redux DevTools 강력
적합 규모 중소규모 전역 상태 대규모 복잡 상태
선택 이유 서버 상태는 TanStack Query가 담당하므로, 클라이언트 전역 상태 범위가 좁음. Zustand의 간결함이 적합.

5. 결제 (Payment)

토스페이먼츠 (TossPayments) 결제 PG · 확정

국내 사용자 UX 최적화, 간편한 JavaScript SDK, 웹훅(Webhook) 기반 서버 사이드 결제 완료 확인 지원.

권한 부여
  • 결제 완료 이벤트: 웹훅(Webhook)으로 Spring 서버가 수신 (POST /api/v1/payments/webhook)
  • 웹훅 수신 후: payments 테이블 상태를 COMPLETED로 업데이트
  • 동시에: chat_rooms 자동 생성, 문진표 열람 권한 부여
  • 중복 웹훅 방어: pg_transaction_id 유니크 제약으로 멱등성 보장
  • 부분 환불·강제 환불은 어드민 전용 API로 처리

대안 비교 — 토스페이먼츠 vs 포트원(iamport)

항목 토스페이먼츠 (선택) 포트원(iamport)
국내 UX 최적화 선택 양호
SDK 품질 현대적, TypeScript 지원 다양한 PG 지원
웹훅 지원 지원 지원
PG 다양성 토스페이먼츠 단일 여러 PG사 선택 가능
선택 이유 MindFit은 국내 서비스 집중, 단일 PG로 충분. 토스페이먼츠의 SDK 개발 경험이 더 직관적.

6. 파일 스토리지 (File Storage)

AWS S3 / NCP Object Storage 오브젝트 스토리지

자격 서류(PDF/이미지), 프로필 사진 등 정적 파일을 서버와 분리하여 저장. Pre-signed URL 방식으로 서버 부하 최소화.

  • 파일 업로드 방식: 클라이언트가 Spring 서버에 Pre-signed URL 요청 → S3에 직접 PUT 업로드
  • 서버는 URL만 DB에 저장, 대용량 파일이 서버를 통과하지 않음
  • 파일 크기 제한: 자격 서류 5MB, 프로필 사진 2MB
  • MIME 타입 검사: 업로드 전 Spring 서버에서 허용 타입 검증
  • 버킷 공개 정책: 프로필 사진 Public, 자격 서류 Private (Pre-signed URL로만 접근)
  • NCP Object Storage: AWS S3 호환 API 제공, 국내 규정 준수 시 대안

7. 개발 환경 (Dev Environment)

로컬 개발 Docker Compose
  • MySQL 8.x 컨테이너 로컬 실행
  • 환경변수 .env 파일 분리 (DB 패스워드, JWT Secret 등)
  • 프론트: npm run dev (Vite HMR)
  • 백엔드: ./gradlew bootRun
CI/CD (추후) GitHub Actions
  • PR 생성 시 빌드·테스트 자동 실행
  • main 브랜치 병합 시 AWS EC2 자동 배포
  • AWS EC2 (애플리케이션) + RDS MySQL (DB)
  • S3 버킷 분리 (운영/스테이징)
개발 우선순위: 로컬 Docker 환경 → 기능 개발 → GitHub Actions CI → EC2/RDS 배포 순서로 진행. 초기에는 배포 환경보다 기능 구현에 집중.
