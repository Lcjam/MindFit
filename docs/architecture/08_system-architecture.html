GET /api/v1/payments/sessions/{sessionId} 세션별 결제 내역 조회 CLIENT COUNSELOR
Webhook 멱등성 처리 절차 — POST /api/v1/payments/webhook
1
중복 확인 — pg_transaction_id로 payments 테이블 SELECT 조회
2
이미 존재하는 경우 → 즉시 200 OK 반환 (재처리 없음). 토스페이먼츠는 200 수신 시 재전송을 중단하므로 무한 재전송 루프가 차단됩니다.
3
존재하지 않는 경우 → 단일 트랜잭션(@Transactional) 내에서 아래 3가지를 원자적으로 처리:
  • a. payments 레코드 INSERT (pg_transaction_id 유니크 제약이 동시 요청 경쟁 조건 방어)
  • b. sessions.status = CONFIRMED 로 업데이트
  • c. chat_rooms 레코드 생성 + questionnaires.is_accessible = true 설정
4
처리 완료 → 200 OK 반환. 트랜잭션 실패(DB 오류 등) 시에는 200이 아닌 5xx를 반환하여 토스페이먼츠가 재전송하도록 유도.
설계 근거: pg_transaction_id 유니크 인덱스만으로는 두 번째 Webhook이 INSERT 실패 → 500 반환 → 토스페이먼츠 재전송 루프로 이어질 수 있습니다. 사전 SELECT 조회(위 1·2단계)를 통해 정상 중복은 200으로 조기 차단하고, 실제 오류만 5xx로 분리하는 것이 올바른 멱등성 처리입니다.

5. WebSocket 상세 설계 (WebSocket Design)

프로토콜 스택
  • 전송 레이어: WebSocket (RFC 6455)
  • 메시징 프로토콜: STOMP 1.2 (Simple Text Oriented Messaging)
  • 폴백: SockJS (HTTP Long-polling, Server-Sent Events)
  • 클라이언트 라이브러리: @stomp/stompjs + sockjs-client
  • 서버: Spring WebSocket + Spring Messaging
채널 구성
  • 핸드셰이크 엔드포인트: /ws
  • 애플리케이션 목적지 접두사: /app
  • 브로커 접두사 (구독): /topic
  • 구독 채널: /topic/chat/{roomId}
  • 메시지 전송: /app/chat/{roomId}/send

연결 및 메시지 송수신 플로우

1
핸드셰이크 & JWT 인증
클라이언트가 /ws 엔드포인트에 SockJS 연결 요청. 헤더에 Authorization: Bearer {JWT} 포함. Spring Security HandshakeInterceptor에서 JWT 검증 → 인증 실패 시 연결 거부(401).
2
채팅방 구독
채팅방 UI 진입 시 클라이언트가 /topic/chat/{roomId} 채널 구독. 서버는 해당 roomId에 접근 권한(채팅방 멤버)이 있는지 검증.
3
메시지 전송
클라이언트가 /app/chat/{roomId}/send로 STOMP 메시지 발송. 페이로드: { content: string, messageType: "TEXT" | "LINK" | "FILE", fileUrl?: string }
4
서버 처리 & DB 저장
Spring @MessageMapping 컨트롤러에서 수신. messages 테이블에 즉시 저장 (chat_room_id, sender_id, content, created_at). 저장 성공 후 응답 DTO 구성.
5
브로드캐스트
/topic/chat/{roomId} 구독자 전원에게 메시지 브로드캐스트 (messagingTemplate.convertAndSend()). 응답 DTO: { id, senderId, senderName, content, messageType, createdAt }
6
채팅방 자동 생성 (채팅방 생성 시점)
결제 완료 웹훅 수신 시 Spring 서버에서 chat_rooms 레코드 자동 삽입. 생성 시점: payments.status = COMPLETED 업데이트 직후 동일 트랜잭션 내에서 처리.

메시지 이력 복원

  • 채팅방 입장 시 REST API GET /api/v1/chat-rooms/{id}/messages?page=0&size=50로 이전 메시지 조회
  • messages(chat_room_id, created_at) 복합 인덱스로 페이지네이션 성능 보장
  • 최신 메시지부터 역순 로드 (무한 스크롤 상향 방식)
  • WebSocket 연결 이후의 신규 메시지만 실시간 수신

6. 보안 고려사항 (Security)

A
JWT 토큰 보안
  • HTTPS 강제 적용 (HTTP → HTTPS 리다이렉트)
  • Access Token: 30분 (짧은 TTL로 탈취 피해 최소화)
  • Refresh Token: HttpOnly Cookie 저장 (JS 접근 차단, XSS 방어)
  • Refresh Token 재사용 탐지(RTR) 구현 권장
Q
문진표 접근 제어
  • 조건 1: 해당 세션의 결제 상태가 COMPLETED이어야 함
  • 조건 2: 요청자가 해당 세션의 담당 상담사 본인이어야 함
  • 이중 검증을 서비스 레이어에서 수행 (API + 서비스 레이어 중복 체크)
  • 문진표 내용은 개인 민감정보 — 향후 컬럼 수준 암호화 검토
F
파일 업로드 보안
  • MIME 타입 화이트리스트 검사 (Content-Type 헤더 + 매직 바이트)
  • 파일 크기 제한: 자격 서류 5MB, 프로필 사진 2MB
  • 자격 서류 S3 버킷 Private 설정 (Public URL 직접 접근 불가)
  • 파일명 UUID 재생성 (경로 순회 공격 방지)
P
결제 웹훅 보안
  • 토스페이먼츠 웹훅 서명 검증 (X-Webhook-Signature)
  • 멱등성: 사전 SELECT → 중복이면 200 즉시 반환, 신규이면 트랜잭션 처리 (상세 절차는 API 섹션 참조)
  • pg_transaction_id 유니크 제약으로 동시 요청 경쟁 조건(race condition) 방어
  • 웹훅 수신 후 토스페이먼츠 API에 재확인 요청 (결제 진위 이중 검증)
W
WebSocket 보안
  • 연결 시 JWT 검증 (HandshakeInterceptor)
  • 채팅방 구독 시 멤버 권한 재검증
  • 메시지 전송 시 채팅방 소속 여부 확인
  • Rate Limiting: 초당 메시지 전송 횟수 제한 검토
G
개인정보보호
  • 개인정보보호법 준수 — 상담 관련 민감 데이터 암호화 저장 검토
  • 상담 내용(채팅) 보존 기간 정책 수립 필요
  • 비밀번호: BCrypt 해싱 (cost factor 12+)
  • 로그에서 개인식별정보(PII) 마스킹 처리
중요: 심리상담 플랫폼 특성상 문진표, 채팅 내용은 일반 서비스 대비 고수준의 데이터 보호가 요구됩니다. 서비스 론칭 전 개인정보보호법 적합성 검토 및 데이터 보호 책임자(DPO) 지정을 권장합니다.
